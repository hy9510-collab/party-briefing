# 한눈에 보는 정치 브리핑 — 기획·설계 문서

> 이 문서는 **"무엇을 어디에 두는가"의 기준**입니다. 코드보다 먼저 봅니다.
> 사이트에 뭘 추가할지, 데이터를 어디에 쓸지 헷갈릴 때 이 문서로 결정합니다.
> 최초 작성: 2026-07-01 · 상태: 초안(로드맵 진행하며 갱신)

## 1. 목적과 대상

대통령실·정부·국회·정당·경기도의 **정책과 이슈를 한 화면에서** 보는 사이트.

- **핵심 가치:** 흩어진 공식 게시판을 매일 한곳에 모아 "오늘 뭐가 있었나"를 빠르게 파악.
- **대상:** 경기도의회에서 근무하는 실무자(내부용). 시민 공개용이 아니므로 전문 용어·기관명은 그대로 써도 된다. 대신 **의정 활동 지원에 실제로 쓰이는 정보**(국회·타 지방의회 동향, 정책·조례)를 우선한다.
- **범위:** 중앙(대통령실·정부·국회·정당) + **경기도 집행부**(도지사·시군). **경기도의회 자체(의안·조례·회의)는 이 사이트에 넣지 않는다** — 별도 사이트에서 다룸(`ordinance-automation` 등). §9-2 참고.
- **원칙:** 지어내지 않는다. 수집이 안 되는 게시판은 **링크로 안내**만 하고 가짜 제목을 만들지 않는다(현행 스크래퍼 정책 유지).

## 4. 데이터 모델 (목표)

**지금:** 데이터가 JS객체(`data.js`)·마크다운(`.md`)·JS박힌JSON(`council.js`) **세 형식**으로 흩어짐 → 뭘 어디서 고칠지 모름.
**목표:** 전부 `data/` 폴더의 **JSON 한 형식**으로 통일. 코드는 `fetch`로만 읽는다.

| 목표 파일 | 담는 것 | 지금 어디에 있나 | 갱신 주체·주기 |
|---|---|---|---|
| `data/president.json` | 대통령실 조직·위원회 | `data.js` `window.PRESIDENT` | **손** / 인사 시 |
| `data/government.json` | 총리·부처 장관 | `data.js` `window.GOVERNMENT` | **손** / 개각 시 |
| `data/assembly.json` | 국회 의장·상임위·교섭단체 | `data.js` `window.ASSEMBLY` | **손** / 원구성 시 |
| `data/parties.json` | 정당 지도부·게시판 | `data.js` `window.PARTIES` | **손** / 전당대회 시 |
| `data/gyeonggi.json` | 도지사·31개 시군 | `data.js` `window.GYEONGGI`,`GYEONGGI_EDU` | **손** / 선거·인사 시 |
| `data/council.json` | 시군의회 당선인 명단 | `council.js` `window.GG_COUNCIL` | **스크래퍼(1회성)** / 선거 후 |
| `data/briefing.json` | 오늘의 이슈·기관별 자료 | `정당정책_오늘내용.md` | **스크래퍼** / 매일 07:30 |

> 데이터 형식만 바꾸는 것이라 화면 결과물은 그대로. `.md`→`.json` 전환 시 `app.js`의 `parseDaily()`(마크다운 파싱) 부분이 사라지고 `fetch(json)`으로 단순해진다.

## 5. 목표 파일 구조

```
정당정책_공개사이트/
├─ index.html          # 껍데기만 (CSS는 밖으로)

│     ├─ home.js  org.js  government.js  assembly.js
│     ├─ gyeonggi.js  briefings.js  detail.js

│  ├─ parties.json  gyeonggi.json  council.json
│  └─ briefing.json   # 스크래퍼가 매일 덮어씀

│  └─ update_briefing.py  parse_council.py
```

> **주의(과하게 쪼개지 말 것):** 정적 사이트다. CLAUDE.md의 "단순함 우선"에 따라, `render/`를 파일 8개로 나누는 게 부담이면 **`app.js` 하나 + CSS 분리 + data JSON화**까지만 해도 목적의 80%는 달성된다. 렌더 분리는 선택.

## 6. 갱신 파이프라인

```
[매일 07:30 KST] GitHub Actions
   → scraper/update_briefing.py 실행 (각 게시판 수집)
   → data/briefing.json 덮어쓰기
   → 변경 있으면 자동 커밋·푸시
   → Vercel이 git push 감지 → 자동 재배포
```

- **손으로 고치는 데이터**(§4의 "손")는 이 파이프라인이 **건드리지 않는다.** 사람이 직접 `data/*.json` 편집 후 커밋.
- "🔄 지금 업데이트" 버튼: 실제 재수집이 아니라 로컬 데이터 다시 읽기. 라벨을 **"최신 불러오기"**로 바꾸거나, 진짜 즉시 수집이 필요하면 `workflow_dispatch`를 부르는 방식(복잡도↑)으로 확장.

## 9. 확인 필요 / 열린 질문

1. ~~대상이 누구인가?~~ **확정(2026-07-01):** 경기도의회 실무자(내부용). 시민 공개용 아님.
2. ~~경기도의회 섹션 추가~~ **정리(2026-07-01):** 경기도의회(의안·회의·조례)는 **별도 사이트로 이미 제작됨**(→ `ordinance-automation` 등) → **이 사이트 범위에서 제외.** 이 사이트의 "경기도"는 집행부(도지사·31개 시군)만 다룬다. 남은 결정은 두 사이트를 서로 링크할지 여부뿐.
3. **경기도·교육청에 "오늘 자료"를 넣을까?** 지금은 조직 정보만. 경기도청 보도자료·도의회 소식을 매일 수집 대상에 넣을지.
4. **수집 범위 확장?** 국회는 현재 링크만. 의안정보시스템·열린국회 Open API로 실제 계류 법안을 끌어올지(상위 목표인 "조례 자동화"와 연결됨). 도의회 실무자에게 특히 유용.
5. **정리 강도:** 로드맵 5단계(app.js 분리)까지 갈지, 데이터 JSON화(2단계)까지만 하고 멈출지.
6. **다른 프로젝트와의 관계:** `gyeonggi-gongmuwon-info`, `공무원_업무정보.html`, `complaint-assistant`와 주제가 겹침 — 통합할지 별개로 둘지.
